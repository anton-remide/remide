# Дизайн-система проекта (аудит и Overview v0)

Дата: 2026-03-12  
Проект: RemiDe (`vite` + `react`)  
Актуальность: синхронизировано с `main` (commit `6a3e334`)

## 1) TL;DR (как устроено сейчас)

- Визуал собирается из трех источников: **Bootstrap CSS**, **кастомный CSS** и **inline-стили** в React.
- Главный файл со стилями и токенами: `src/styles/app.css` (токены + большая часть UI-классов `st-*`).
- В проекте много готовых UI-паттернов в `st-*`: в `app.css` уже около **877** селекторов, начинающихся с `.st-`.
- `Bootstrap` подключен глобально, но bootstrap-grid (`row/col/g-*`) в JSX больше не используется (Phase 1). Остаются только отдельные базовые/утилитарные стили, пока импорт не удален.
- Для JS-логики ширины введен единый реестр: `src/constants/breakpoints.ts` (используется вместо "магических" чисел в коде).
- `Inline` все еще много: **466** вхождений `style={{...}}` в `src/` (в **26** файлах), из них **155** используют `var(--...)` (в **18** файлах).
- Есть отдельная "тема" в коде: `src/theme.ts` (цветовые мапы для бейджей/карт), это отдельный слой от CSS-токенов.
- После последних изменений в `main` добавились крупные UI-блоки: **paywall gate**, **floating CTA**, обновленный **landing hero** и блоки аудитории.

## 2) Источники правды и приоритеты (что "побеждает")

Фактический порядок подключения (см. `src/main.tsx`):

1. `bootstrap/dist/css/bootstrap.min.css`
2. `src/styles/app.css`

Дополнительно, в `WorldMap` подключается `maplibre-gl/dist/maplibre-gl.css`.

Практический приоритет (от сильного к слабому):

1. **Inline-стили** (`style={{...}}`) — почти всегда сильнее обычных классов.
2. **`app.css`** — подключен после Bootstrap, поэтому при равной специфичности перекрывает его.
3. **Bootstrap** — пока подключен как внешний CSS-слой, но не как grid-система.
4. **Браузерные дефолты**.

Что важно помнить:

- Если "что-то выглядит странно", почти всегда это смесь Bootstrap + `app.css` + inline.
- Если "цвета статусов/карты выбиваются", это обычно зона `src/theme.ts`, а не `:root` токены.
- Для JS-брейкпоинтов больше не используем "голые числа": берем значения из `src/constants/breakpoints.ts`.

### Единый реестр брейкпоинтов (Phase 1)

- Источник правды для JS: `src/constants/breakpoints.ts`.
- Используемые значения: `1280 / 1024 / 991 / 767 / 640 / 600 / 520 / 480 / 375`.
- Для CSS эти же пороги зафиксированы в `@media` правилах `src/styles/app.css`.

## 3) Токены (foundations): что есть и где менять

Базовые токены живут в первом `:root { ... }` внутри `src/styles/app.css`.

### Основные группы токенов

- **Шрифты**: `--font1` (DM Sans), `--font2` (Doto).
- **Core-цвета**: `--color-primary`, `--color-bg`, `--color-surface`, `--color-border`, `--color-text-main`, `--color-text-secondary`, `--color-accent`.
- **Семантические алиасы (compatibility)**: `--white`, `--black`, `--text`, `--text-muted`, `--border`, `--accent`, `--bg-light`.
- **Радиусы**: `--radius-pill`, `--radius-lg`, `--radius-md`, `--radius-sm`.
- **Тени**: `--shadow-soft`.
- **Размеры иконок**: `--icon-xs/sm/md/lg`.
- **Layout-метрики**: `--header-height`, `--header-current-height`, `--header-content-gap`, `--header-content-offset`, `--top-banner-height`.
- **Landing-переменные** (`--landing-*`) задаются локально в `st-landing-v2`, а не в `:root`.

### Token map (быстрый словарь)

| Что меняем | Где менять |
|---|---|
| Базовые бренд-цвета, текст, фон | `:root` в `src/styles/app.css` (`--color-*`) |
| Типографику и шрифтовые пары | `:root` (`--font1`, `--font2`) + импорт Google Fonts |
| Скругления и базовые размеры интерфейса | `:root` (`--radius-*`, layout vars) |
| Landing-специфичные цвета/гаттеры | `.st-landing-v2 { --landing-* }` |
| Статусные цвета (бейджи/карта) | `src/theme.ts` (цветовые мапы) |

### Где менять "главное"

- Основной фон: `--color-bg`.
- Основной текст: `--color-text-main` и `--color-text-secondary`.
- Акцентный цвет: `--color-accent`.
- Скругления: `--radius-*`.
- Шрифты: `--font1`, `--font2` + импорт Google Fonts вверху `app.css`.

Примечание по актуальности: в последних коммитах `main` **`:root` токены не менялись**, менялись в основном компонентные блоки и страницы.

## 4) Компоненты/паттерны: что есть (через `st-*`)

Дизайн-система в текущем состоянии — это в первую очередь слой классов `st-*` в `src/styles/app.css` + точечные inline-исключения.
`st-*` — это CSS-контракты визуальных паттернов (не React-компоненты).  
Пример: `st-btn` = контракт кнопки, `st-card` = контракт карточки, `st-paywall-gate` = контракт paywall-блока.

Часто используемые паттерны сейчас:

- **Layout/страницы**: `st-page`, `st-landing-v2`, `st-hero`.
- **Header/Footer/Banner**: `st-header*`, `st-footer*`, `st-top-banner`.
- **Кнопки/ссылки**: `st-btn`, `st-btn-outline`, `st-btn-sm`.
- **Карточки/контейнеры**: `st-card`, `st-info-card`, `st-feature-card`.
- **Таблицы и data view**: `st-table`, `st-info-row`, `st-info-label`, `st-info-value`.
- **Paywall слой**: `st-paywall-gate*`, `st-floating-cta*`, плюс blur-классы `st-blur-*`.
- **Landing/Pricing новые блоки**: `st-audience-*`, `st-pricing-*`, `st-landing-*`.
- **Карта**: `st-map-container*` + внешний CSS от MapLibre.

Если менять общий вид кнопок/карт/таблиц/хедера, почти всегда правка в `src/styles/app.css`.

## 5) Правила проекта (v0), чтобы не усложнять

1. Сначала ищем токен, потом хардкод (`var(--...)` предпочтительнее `#HEX`).
2. Grid/layout делаем через `st-*` (CSS Grid/Flex), не через bootstrap-grid классы.
3. Новые inline-стили запрещены. Исключение: только динамические случаи с явной пометкой `inline-dynamic-ok: <reason>`.
4. Новые классы называем с префиксом `st-`, чтобы не конфликтовать с Bootstrap.
5. Цвета статусов/режимов карт и бейджей держим в `src/theme.ts` (пока не принято решение об объединении).
6. Глобальные изменения брендинга (фон/радиусы/типографика) делаем через `:root` токены.
7. Изменения в монетизации/UI-потоке (paywall, pricing, floating CTA) проверяем как отдельный слой, а не только как "еще один блок страницы".
8. Брейкпоинты в JS берем только из `src/constants/breakpoints.ts`, чтобы не расходиться с CSS.
9. Перед каждой новой задачей обновляем локальный `main`, потом работаем в отдельной ветке.
10. Если inline действительно нужен для динамики, добавляем рядом пометку: `inline-dynamic-ok: <reason>`.

## 6) Точки риска (текущее смешение подходов)

- `app.css` остается монолитом: токены + reset + большинство компонентных стилей в одном файле.
- Inline-стили масштабны: 466 вхождений `style={{...}}`, поэтому некоторые страницы трудно глобально переоформлять.
- Два "мира" цветов: CSS-токены vs `src/theme.ts` для статусов/карт.
- Bootstrap импорт пока не удален (Phase 2), поэтому часть каскада все еще может влиять на итоговый вид.
- Док быстро устаревает по цифрам, потому что `main` активно меняется (особенно landing/pricing/paywall).
- В `CLAUDE.md` есть ссылка на `/Theme SquareType/`, но самой папки в текущем репозитории нет.

## 7) Как обсуждать изменения (для non-dev): токен / компонент / страница

Используйте этот чек-лист:

| Ситуация | Что менять |
|---|---|
| Изменение влияет на весь продукт (бренд, фон, текст) | Токены в `:root` |
| Изменение влияет на тип элемента (все кнопки/карточки/таблицы) | `st-*` класс в `app.css` |
| Изменение только на одном экране | Локальный `st-*` блок экрана |
| Нужен цвет статуса/регима/карты | `src/theme.ts` |
| Нужен inline | Только динамика + `inline-dynamic-ok` |

1. Это про весь продукт (фон, шрифт, общий цвет, радиусы)?
   - Да: меняем токены в `:root` (`src/styles/app.css`).
2. Это про тип элемента (все кнопки, все карточки, все таблицы)?
   - Да: меняем соответствующий `st-*` класс в `src/styles/app.css`.
3. Это про конкретный экран/блок (только Landing, только Pricing, только Detail)?
   - Да: меняем локальные `st-*` блоки страницы, стараясь не ломать глобальные паттерны.
4. Это про статус/бейдж/цвет карты?
   - Да: проверяем `src/theme.ts`.
5. Это про блокировку контента/конверсию (paywall, CTA, pricing)?
   - Да: проверяем вместе `st-paywall-*`, `st-floating-cta-*`, `usePaywall`/`useSubscription`, а не только CSS одной страницы.
